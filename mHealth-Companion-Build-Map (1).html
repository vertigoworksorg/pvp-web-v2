01

What I'm adding

Three things on top of the existing app — everything else is reused.
1

Secure agent backend

Node.js CommonJS serverless, extends the existing fitbit-api/ Vercel pattern. No TS, no build step — plain .js functions. All keys server-side.

2

RAG profile context

Structured profile assembly + a lightweight semantic memory store with top-k retrieval. Pragmatic for one week — no managed vector DB.

3

Actions framework

Model proposes → backend validates schema → executes. Extensible registry, surfaced through a Flutter command-deck chat UI.

03

Don't break these

The four invariants. Easy to violate under a one-week crunch — guard them.

Non-negotiable invariants

health data · cancer survivors
1

No LLM/provider keys in the Flutter client. The app only ever talks to the backend.

2

Identity = verified Firebase ID token, never the model. Current /api/ai/chat takes {messages} with no auth → add token verification, bind every read/write to the verified uid.

3

Owner-scoped reads only, matching existing firestore.rules. A user can never retrieve another's data.

4

Model proposes; backend decides. Schema-validate before execute; nothing impactful runs without user confirmation in the UI.

08

Trade-off levers

What slips first if the week tightens — neither blocks acceptance.
Not a gate

Streaming replies

Sequenced last on Day 3. Token streaming through a serverless proxy plus a RAG + tool round-trip can eat a day. If it doesn't fit, replies ship as fast complete responses and streaming moves to Improvements.

Decide at M2

FCM push nudge

The designated lever. If the week's at risk, defer the basic push at M2 → it becomes a quoted Improvement. In-app proactivity always ships; deferral doesn't block M3 or final payment.

09

Watch & verify

The tightest part of the week, and the hooks I build toward.

Tightest risk

The two impactful actions — adjust workout plan & set goal — need real write paths into WorkoutRoutineModel / GoalsModel plus confirm UI, not stubs. Scoped Day 4–5. The generator-re-run framing is the de-risk.

Acceptance hooks

  • RAG: seeded profile + 1 memory → one reply references both (repeatable).
  • Actions: all 5 run, schema-validated, confirm on impactful; invalid rejected.
  • Proactive: in-app check-in greets, asks, flags a consistency gap.
  • Deployed: backend on client hosting; app talks e2e; no keys in client.
